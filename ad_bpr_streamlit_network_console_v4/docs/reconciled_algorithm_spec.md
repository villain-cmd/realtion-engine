# 広告運用計算機 v2 — 矛盾補正済みアルゴリズム仕様

## 1. 目的

本システムは、楽天RPPおよびYahooショッピング広告の実績CSVを読み込み、商品単位を主判定として粗利最大化のためのCPCを計算する。外部AI API、常設DB、自動媒体操作は必要としない。

運用は次の循環で成立する。

```text
前回 state_bundle.zip（任意）
+ 最新の実績CSV
+ 最新の入札設定CSV
+ 商品マスタCSV
→ データ品質検査
→ 短期/長期指標生成
→ 粗利・ROASガードによるCPC計算
→ 異常判定
→ GUIで承認/修正/ロック/停止
→ 入稿CSV・ロールバックCSV・次回 state_bundle.zip
→ 次回CSV投入時に継続
```

## 2. 回答の優先順位

回答間に矛盾がある場合は次の順に解決した。

1. 今回の最新指示「計算機として使い、異常時は手動変更し、その後も再投入で動く」
2. 回答末尾の「初期MVPで固定すべき質問」
3. 個別設問の回答
4. 数理的に成立しない、または事故を誘発する回答は安全側へ補正
5. 値が未指定の項目は設定可能な暫定値を置き、コードへ固定しない

## 3. 最適化指標

### 3.1 主目的

主目的は商品単位の広告貢献粗利最大化とする。

```text
広告前粗利      = 属性売上 × 標準粗利率
その他販促費    = 属性売上 × その他販促費率
広告貢献粗利    = 広告前粗利 - その他販促費 - 広告費
```

「粗利」と「ポイント・クーポン・アフィリエイト等を販促費に含める」という回答を両立させるため、会計上の粗利と広告判断用の貢献粗利を分離した。

ROASは主目的ではなく、手動設定するガード条件と評価指標である。

### 3.2 CPCの二重ガード

```text
平滑化CVR = (注文数 + 事前CVR × 事前強度) / (クリック数 + 事前強度)
期待売上/クリック = 平滑化CVR × 期待AOV
広告前貢献粗利/クリック = 期待売上/クリック × (粗利率 - その他販促費率)

粗利ガードCPC = 広告前貢献粗利/クリック × (1 - 安全余白率)
ROASガードCPC = 期待売上/クリック ÷ 目標ROAS倍率
生の推奨CPC = min(粗利ガードCPC, ROASガードCPC)
```

粗利率未登録時は既定粗利率で試算できるが、既定では入稿承認不可とする。

## 4. 判定粒度と親子制約

- 主判定単位は商品。
- SKU・バリエーションは親商品へ集約。
- キーワードは商品配下の副判定。
- 商品が停止なら配下キーワードも停止候補。
- 商品が減額なら配下キーワードの増額は禁止。
- 商品CPCを親の上限・方向性として優先する。
- 商品別レポートとキーワード別レポートを店舗KPIで合算しない。店舗KPIは商品別を優先し、二重計上を防ぐ。

## 5. 現在CPCの定義

制御基準と評価値を分離する。

```text
制御基準（current_bid）優先順位
1. 有効な手動LOCK/BASELINE
2. 今回アップロードされた媒体設定CSVの登録CPC
3. 前回state_bundleの適用CPC
4. 実績レポートの登録入札値
5. 実績CPC（最終フォールバックのみ）
6. 共通最低CPC
```

実績CPCはオークション結果であり、媒体へ設定する値ではないため、通常の制御基準にはしない。

## 6. 観測期間

- 主判定は直近7日。
- 長期比較は28日。
- 複数期間CSVがあれば、終了日が最新で7日・28日に最も近いスナップショットを採用。
- 1ファイルしかない場合は短期・長期を同一値として扱い、トレンド補正を無効化。
- 前年同期・前週同曜日は将来のML特徴量として保持するが、MVPルール判定の必須入力にはしない。
- 楽天720時間成果は直近データほど未成熟であるため、成熟度を信頼度として表示する。未成熟時の減額は自動承認せず、増額は常に手動承認。

## 7. データ量不足と新商品

- 通常判定の最低クリック数は10。
- 新商品専用の強制探索予算は設けない。
- 販売開始14日未満は観測状態。
- 10クリック未満は観測状態。
- クリック0は効果不良と断定できないため、露出・需要・適格性の手動確認へ送る。
- 注文0かつ50クリック以上は停止または最低CPC候補。
- 新商品の「50クリック探索」「予算5%」は、強制探索をしないという最終MVP回答と競合するため不採用。

## 8. ベイズ平滑化

事前分布は階層的に決める。

1. キーワードは親商品のCVRを優先。
2. 商品は商品群・カテゴリ平均。
3. 不足時はモール×粒度平均。

事前強度の既定値は20クリック。GUI・policyで変更可能。

増額判定では平均CVRだけでなく、ベータ事後分布の片側80%下限を使う。

```text
CVR下限 × AOV × (粗利率 - その他販促費率) > 推奨CPC
```

この条件を満たさない増額候補は据え置く。

## 9. CPC制約

既定値:

- 最低CPC: 24円
- 最高CPC: 100円
- 最大増額率: 50%
- 最大減額率: 50%
- 1回最大変更額: 30円（回答に金額がなかったため暫定値）
- デッドバンド: max(現CPCの10%, 3円)
- 円未満: 切り上げ

回答の「差が50%未満なら据え置き」は、最大変更率50%と組み合わせるとほぼ全提案が停止するため、デッドバンド10%へ補正した。

停止は100%減額ではなく別の状態 `STOP_OR_MIN_CPC` とし、媒体仕様が確定するまでは最低CPC出力を安全な既定動作とする。

## 10. 状態優先順位

上から先に適用する。

1. データ品質BLOCK
2. 手動除外
3. 在庫0、終売、予約販売、無効などのハード停止
4. 手動CPCロック
5. 異常検知
6. 前回変更から7日未満の効果観測
7. 粗利率未登録
8. 新商品観測
9. クリック0の手動確認
10. クリック不足
11. 注文0停止閾値
12. 単位採算不成立
13. デッドバンド
14. 減額
15. 増額
16. 据え置き

## 11. 承認

- 増額: 必ず手動承認。
- 減額: 条件を満たせば初期値ACCEPT。
- 停止候補: 条件を満たせば初期値ACCEPT。
- 粗利仮定値、アトリビューション未成熟、重複入力、データ品質異常は自動承認しない。
- 「自動承認」は媒体へ自動送信する意味ではなく、ダウンロードファイルへ反映する初期選択を意味する。
- Streamlitから媒体へ直接入稿しない。

## 12. 異常検知と手動介入

前回状態に同一商品・キーワードの履歴があり、双方が20クリック以上の場合に比較する。

既定シグナル:

- CVRが50%以上低下
- ROASが50%以上低下
- 実績CPCが50%以上上昇

2つ以上が同時発生すると `ANOMALY_MANUAL_INTERVENTION` とし、その行の自動出力を止める。

GUIで次の操作を行う。

- `MODIFY`: 手動CPCを今回だけ適用
- `LOCK`: 手動CPCを次回以降も固定
- `FORCE_STOP`: 最低CPC/停止候補として固定
- `REJECT`: 提案を不採用

異常後の手動値は `planned_settings` と手動ポリシーへ保存される。次回の媒体設定CSVで一致した時点に `applied_settings` へ確定し、その適用時点から7日間観測した後、通常ロジックへ復帰する。

## 13. 変更後の観測

回答には「再変更を禁止しない」と「7日で効果判定」が併存していた。因果を壊さないため次へ補正した。

- 通常は変更後7日間、再提案を観測状態にする。
- 在庫0、終売、重大データ異常は観測期間を上書きできる。
- operatorが手動変更した場合も同じ。
- 価格・クーポン・商品ページ変更は別レジームとして記録する。計算自体は継続するが、前回入札変更の効果評価は交絡ありとして学習から除外できる設計にする。

## 14. 状態バンドル

Streamlit Community Cloudのローカルディスクは永続前提にできないため、状態をZIPで持ち回る。

`state_bundle.zip`:

- `state_manifest.json`: スキーマ版、最終Run ID
- `policy.json`: 今回の設定値
- `applied_settings.csv`: 媒体設定CSVで確認済みのCPC
- `planned_settings.csv`: 出力済みだが媒体反映未確認のCPC
- `manual_overrides.csv`: LOCK・FORCE_STOP等
- `action_history.csv`: 推奨、承認、手動値、適用日、効果判定期限
- `metric_history.csv`: 次回異常判定用の実績スナップショット
- `run_history.csv`: 入力hash、品質、行数、重複実行

次回にstate_bundleがなくても最新設定CSVを基準に計算できる。ただし異常比較・変更履歴・ロックは引き継げない。

## 15. 外部手動変更の検出

前回state_bundleの適用CPCと今回設定CSVの登録CPCが1円以上異なる場合、媒体画面等で手動変更されたとみなす。

- 今回設定CSVを正とする。
- `EXTERNAL_MANUAL_CHANGE` として履歴へ保存。
- その日時から7日観測。
- 次回以降の推奨は新しいCPCを基準にする。

これにより、アプリ外で変更しても次回データ投入で壊れない。

## 16. データ品質

### Run BLOCK

- 必須列不足
- 商品ID欠損
- クリック、費用、売上、注文の負数
- 必須値の不正行が1%超
- 前回比で行数が20%以上急減
- ファイル種別誤り

負数・空欄の広告費を0へ黙って置換しない。0円と欠損は意味が異なるため、エラー行として除外し、1%を超えたらRunを止める。

同一入力hashの再実行は計算可能だが、自動承認を外し、重複入稿を防ぐ。

## 17. 入稿ファイル

- 楽天の商品CPCとキーワードCPCを別ファイルで生成。
- 商品ファイルは商品ごとに1行。
- キーワードファイルは全キーワード行。
- 未変更行も含む完全ファイル。
- 承認済み変更だけ `コントロールカラム=u` とし、未承認・未変更は元値のまま。
- CP932、CRLF、媒体ヘッダー順。
- 同時にロールバックファイルを生成。
- Yahooは媒体固有CSVが提供されるまで正規化レビューCSVを生成し、adapter差し替えにする。

## 18. 自然売上・イベント・予算

回答に矛盾が多いため、MVP判定への組込みを次のように限定した。

- 自然売上は主判定に使わない。
- 提供された場合はMER・純増性の診断にのみ使う。
- イベントは同一モデル内の特徴量・手動事前戦略として記録する。別モデルにはしない。
- 店舗予算上限が未設定なら予算再配分最適化を実行しない。期待粗利増分順のランキングのみ出す。
- イベント時3倍は予算上限またはイベントカレンダーが設定された場合だけ適用する。

## 19. MLへの移行

### M0: 現在の計算機

- ベイズCVR平滑化
- AOV推定
- 粗利/ROAS二重ガード
- 異常検知
- 手動介入履歴

### M1: 履歴蓄積後

- モール別・商品/キーワード別・カテゴリ特徴量を持つ共通モデル
- CVR: Log Loss/Brier/Calibrationを主評価。MAEのみにはしない。
- AOV/売上: weighted MAE、RMSE、Tweedie等を比較。
- CPC反応: CPC変更率に対する表示・クリック弾力性を学習。
- 再学習は週次、学習履歴は最低90日、推奨180日以上。
- 楽天720時間ラベルは成熟したクリック群だけを教師に使う。最新7日だけでは学習しない。

### モデル採用条件

- 現行ルールと人判断の双方をベースラインにする。
- 粗利改善を主KPIにする。
- 7日は中間監視、採用判定は楽天では最低28日または720時間成熟後。
- 性能悪化時はM0ルールへ戻す。
